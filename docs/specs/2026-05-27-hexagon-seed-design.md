# Design spec — Hexagon seed (Stålberg Variant B), Phase H1

**Date:** 2026-05-27
**Status:** draft — awaiting operator review
**Author:** Kai Denrei (Claude Opus 4.7)

## Scope

- **In scope (build now):** a *single* hexagonal patch generated by a hex-lattice seeder, selectable in the existing 2D view alongside Poisson. Includes the small refactor (H0) to make the seeder pluggable.
- **Forward-compatible (design-only, do NOT build):** incremental patch-adding (H2b) — the operator's chosen eventual "grow" mechanism. The interfaces below must not preclude it.
- **Out of scope:** multi-patch generation/stitching (H2), true infinite streaming (H3), and the 3D second tab. Each gets its own spec later.

## Background

This is Oskar Stålberg's **Variant B** — the hex-lattice seed, the *true* Townscaper look (already documented in `docs/02-grid-algorithm.md` stage 1B; descoped from V1). The operator's reference frames map onto stages we already have: points-in-hexagon → triangulate → dissolve a quad → split leftover triangles into 3 quads → split every quad into 4 → relax.

**Key insight:** the existing pipeline is **seed-agnostic.** `src/grid.js` stages 2–5 (triangulate/filter → merge → subdivide → winding → relax) consume a point set and return a relaxed all-quad mesh; they do not care whether the points came from Poisson or a hex lattice. So H1 is fundamentally: **add a hex seeder and route the same pipeline through it.** The only new code is the seeder and a view generalization.

## Architecture

### Component 1 — `src/hex.js` (new): hexagonal lattice seeder
Pure geometry, no DOM, no RNG (deterministic by construction; the pipeline's `seed` still drives the random dissolve downstream).

- `hexLattice({ rings, spacing = 0.1, center = [0, 0] }) → { points, boundary }`
  - `points`: `[[x,y], ...]` — a triangular point lattice clipped to a hexagonal outline of `rings` rings around `center`. Triangular basis `e1 = (spacing, 0)`, `e2 = (spacing/2, spacing·√3/2)`; a lattice node at axial `(q, r)` sits at `center + q·e1 + r·e2`. Include all `(q, r)` with hex-distance `max(|q|, |r|, |q+r|) ≤ rings`. Ring *k* contributes 6*k* points; total = `1 + 3·rings·(rings+1)` (centered hexagonal numbers: rings 1→7, 2→19, 3→37, 4→61).
  - `boundary`: the indices of the outermost ring's points (hex-distance == rings). **Unused in H1**, exposed now so H2b can later identify the shared edge between adjacent patches at zero extra cost.
- `spacing ≈ 0.1` matches the Poisson `r=0.1` density so the existing relaxation constant `SIDE_LENGTH=0.06` produces comparable cell sizes for both seeders.

### Component 2 — `src/grid.js` refactor (H0): pluggable seeder
- `generateMesh({ seed = 0, seeder = 'poisson', r = 0.1, k = 30, rings = 4, spacing = 0.1 } = {}) → { vertices, quads, seed, seeder }`
  - `seeder === 'poisson'` → existing Bridson path (unchanged; defaults unchanged).
  - `seeder === 'hex'` → `hexLattice({ rings, spacing })`, then the **same** stages 2–5.
- **Backward compatibility is mandatory:** `generateMesh({ seed })` must behave exactly as today (Poisson). The existing 66 tests must pass untouched.
- Internally, extract a `seedPoints(opts)` dispatcher so stages 2–5 stay a single shared code path with no per-seeder branching below the seed step.

### Component 3 — view generalization (`src/main.js`): fit-to-bounds
- Today `fitView()` maps a fixed `[0,1]²` into the canvas. Poisson points live there; a hex patch (world units, centered on origin) does not.
- Change: compute the **bounding box of `mesh.vertices`** whenever a mesh is (re)generated, and fit *that* box into the canvas (centered, aspect-preserved, with margin). For Poisson the bbox ≈ `[0,1]²`, so behavior is unchanged. For hex it frames the hexagon.
- `fromScreen` (the paint inverse transform) must use the same bbox, so click-to-paint keeps working for both seeders.
- Keep the transform behind a small `view` object (bounds + `toScreen`/`fromScreen`). This is the seam a future **pan/zoom camera** (needed for H2) slots into — but H1 ships auto-fit only (no pan/zoom; YAGNI).

### Component 4 — UI (`src/controls.js`, `index.html`): shape selector
- A **shape selector** (segmented control / two radio buttons): `Poisson` | `Hexagon`, styled in the existing dark/amber idiom.
- Contextual sliders: when `Hexagon` is active, show a **Rings** slider (range 2–6, default 4) and hide the Poisson **Density (r)** slider; vice-versa. **Pull rate** and **Iterations** apply to both. Seed readout unchanged.
- Any change live-regenerates with the selected seeder + params.

### Reused as-is (must keep working with the hex mesh)
M2's `halfedge.js` / `dual.js` / `state.js` and click-to-paint operate on any quad mesh, so they work for hex with no change — *provided* the paint inverse transform uses the new bounds (Component 3). Dual cells render only for interior vertices (degree ≥ 3); hex boundary vertices are skipped, identical to Poisson.

## Data flow
`selector + sliders` → `{ seeder, rings|r, pullRate, nIters, seed }` → `generateMesh` → `{ vertices, quads }` → compute bbox → `view = fitView(bbox)` → animate `relaxer.step()` per frame → `drawMesh`; on settle → build `halfedge`/`dual`/fresh `state` → pointer paints through `view.fromScreen`.

## Determinism, edge cases, error handling
- **Determinism:** the lattice is deterministic; the dissolve step consumes the seeded `mulberry32`. `generateMesh({ seeder:'hex', rings:R, seed:S })` twice must be identical.
- **Minimum size:** guard `rings ≥ 2` (rings 0–1 give too few points / a trivial mesh). The selector's slider range enforces this.
- **Delaunay on a regular lattice:** a perfect triangular lattice is cocircular in places, so Delaunator may pick arbitrary diagonals — this is harmless (the dissolve step randomizes merges anyway, and equilateral triangles are already Delaunay). If it ever proves problematic, the fallback is to construct the triangle lattice directly (docs/02 notes hex needs no Delaunay); not expected to be necessary for H1.
- **Boundary slivers:** the hex outline is clean, so the obtuse-triangle filter removes far fewer triangles than for Poisson; acceptable.

## Testing (Node `node:test`, pure logic)
- `hexLattice`: point counts match `1 + 3R(R+1)` for R = 2,3,4; nearest-neighbour distance ≈ `spacing`; `boundary` set size == `6·rings`.
- Pipeline over hex (parametrize the existing invariant suite by seeder): all-quad, watertight, no-NaN/Infinite (pre+post relax), no zero-area quads post-relax, determinism (same seed+rings identical; different differ), relaxation reduces squareness error.
- **Regression:** the existing 66 tests (Poisson default) stay green.
- Visual: headless-Chrome screenshot of the Hexagon mode showing a relaxed hexagonal patch + a painted dual cell.

## Forward-compatibility for H2b (incremental patches) — design-only
These cost ~nothing now and keep the incremental-growth path open:
- `hexLattice` already takes a `center` (so a patch can later be placed at an arbitrary hex-of-hexagons location) and returns `boundary` indices (the shared-edge candidates).
- The mesh is plain `{vertices, quads}` arrays — H2b will dedupe two patches' shared boundary vertices and relax with those pinned. No change needed to the mesh shape.
- The `view` object is the seam for a pan/zoom camera spanning multiple patches.

**Do NOT build now:** patch placement/management, shared-boundary dedup, pinned relaxation, pan/zoom camera, any multi-patch logic. H1 is one centered patch with auto-fit.

## Out of scope (separate specs)
- **H2 / H3** — connected patches and infinite streaming.
- **3D second tab** — WebGL extrude + trilinear test shape.
